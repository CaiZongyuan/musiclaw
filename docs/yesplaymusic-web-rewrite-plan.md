# YesPlayMusic Web 1:1 复刻计划

目标：以 `/root/Projects/Frontend/YesPlayMusic` 的 **Web 端源码、功能、UI 结构与主要交互** 为唯一产品基线，在当前 `TanStack Start + React 19 + TanStack Router + TanStack Query + Tailwind v4` 仓库中完成尽可能接近 **1:1** 的复刻。

范围：

- 仅做 Web
- 暂不考虑 Electron 专属能力
- 第一优先级是 **和原项目对齐**，不是做“更现代”的新产品
- 可以替换技术实现，但不应随意改变信息架构、页面骨架和关键交互

## 一、总原则

### 1. 产品基线以旧仓库为准

以下内容默认以旧仓库源码为第一参考，而不是以当前重写仓库现状为准：

- 路由结构
- 页面区块顺序
- UI 信息层级
- 按钮位置与入口关系
- 播放器行为
- 登录流程
- 空态 / 错误态 / 加载态

### 2. 技术栈可以新，产品表现要旧

允许使用以下替换：

- `vue-router` → `TanStack Router`
- `vuex` → `zustand`
- Vue 组件 → React 组件
- 旧请求封装 → `axios` + server functions / loaders

但替换后要满足：

- 页面职责不漂移
- 用户路径不漂移
- 同名页面尽量保持相同视觉骨架和交互目的

### 3. 非原版能力默认不前置

以下内容如果不是原版 Web 明确已有能力，不应抢占主线：

- 额外演示页
- 过早的 UI 优化
- 自定义信息架构
- 脱离原版的播放器扩展交互
- 先于 parity 的工程抽象

### 4. 完成标准从“能跑”改为“对齐”

以后阶段完成不再只看“接口通了 / 组件写了 / 页面能打开”，而要看：

- 功能是否与旧版一致
- 页面结构是否与旧版一致
- 主路径交互是否与旧版一致
- 明显视觉层级是否与旧版一致

当前仓库已经完成了一批基础设施和数据接入，但主线存在这些偏差：

- 当前进度记录更偏工程实现，不偏 parity 对照
- 路由覆盖仍少于旧版完整路由树
- 还存在 `about`、`demo/*` 这类不属于原版主线的页面
- 已开始推进部分扩展交互，但尚未建立逐页复刻验收表
- 对“首页 / 搜索 / 播放器 / 登录 / Library”是否接近原版缺少显式检查

## 三、新的阶段顺序

### Phase A：建立 parity 基线

目标：先把“要复刻什么”定义清楚，避免继续按感觉推进。

任务：

- 对照旧仓库梳理完整路由树
- 对照旧仓库梳理全局布局与播放器壳体
- 为每个页面建立 parity checklist
- 将 `docs/progress.md` 改为“对齐状态驱动”

交付物：

- 本计划文档
- 更新后的 `docs/progress.md`
- 更新后的 `docs/lessons-learned.md`

完成标准：

- 团队用统一的“原版对照表”而不是自由发挥推进开发

### Phase B：先复刻全局框架

目标：先把用户最先感知到的骨架对齐。

任务：

- 对齐左侧导航栏结构与入口顺序
- 对齐主内容区留白、页宽、滚动方式与全局布局
- 对齐底部播放器结构、主按钮布局和歌词入口
- 对齐歌词层的进入方式和基本表现

完成标准：

- 从任意核心页进入时，整体视觉层级与旧版接近
- 底部播放器主控制区和旧版路径一致

### Phase C：按旧版用户路径逐页复刻

目标：优先复刻原版最常用页面，而不是按当前实现便利度推进。

优先顺序：

1. `/`
2. `/search/:keywords?` 与 `/search/:keywords/:type`
3. `/playlist/:id`
4. `/album/:id`
5. `/artist/:id`
6. `/next`
7. `/login` / `/login/account` / `/login/username`
8. `/library` / `/library/liked-songs`
9. `/settings`
10. `/mv/:id` 与 `/artist/:id/mv`
11. `/new-album`
12. `/explore`
13. `/daily/songs`
14. `/lastfm/callback`

每页都按同一模板验收：

- 路由和入口是否存在
- 页面区块顺序是否一致
- 卡片 / 列表信息密度是否一致
- 核心按钮和跳转是否一致
- 加载态 / 空态 / 错误态是否合理
- 是否能从该页进入原版对应的播放链路

### Phase D：补齐业务细节和边缘能力

目标：在核心页面基本对齐后，再补旧版剩余能力。

任务：

- 喜欢 / 收藏 / 登录态相关细节
- 日推与个性化入口
- MV 播放体验
- Last.fm 回调等边缘路由
- 缓存、持久化与性能优化

完成标准：

- 核心使用路径与原版差异显著收敛
- 剩余差异点被明确记录，而不是隐性遗漏

## 四、Parity 验收方法

每个页面都应在 `docs/progress.md` 中标记为以下三种状态之一：

- `1:1 已对齐`：功能、主要 UI、关键交互都接近旧版
- `功能已到位，UI/交互未对齐`：能用，但仍不是复刻完成态
- `未实现 / 未核对`：不能宣称已完成

额外要求：

- 如果页面只是“能展示数据”，不能直接记为完成
- 如果 UI 是全新布局，即使功能通了，也只能算未对齐
- 如果为了适配 React 而改写实现，需保证用户视角下的主要行为不变

## 五、当前执行约束

从这份文档开始，默认遵循以下约束：

- 先看旧仓库源码，再决定当前仓库怎么写
- 先补 parity checklist，再做具体实现
- 先补缺失原版页面，再考虑增强体验
- 非原版页面不删除也不继续扩展，但不占主线优先级
- 每完成一个阶段，都要明确说明“与原版相比还差什么”
