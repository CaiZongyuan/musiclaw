### H. VIP 误判与试听音源修正

已完成：

- 已确认播放器试听 / 大量 `VIP Only` 的核心根因是：
  - `/song/url` 之前走服务端 `createServerFn`
  - 服务端拿不到浏览器 localStorage 持久化的网易云 `rawCookie`
  - VIP 账号因此会被当成未登录账号取流
- `src/features/player/components/player-engine.tsx` 已改为客户端直接请求 `/song/url`
- 客户端取流已显式开启：
  - `attachNeteaseCookie`
  - `attachRealIp`
- 已补 `freeTrialInfo` 判断，避免播放器继续使用只能试听的音源
- 已新增 `src/lib/music/playability-client.ts`，用于在客户端按当前本地账号态重新计算 `playable / reason`
- 搜索页、歌单页、专辑页、艺人页、每日推荐页现在都会做客户端二次校正，不再只信任服务端首屏的 `VIP Only`
- `src/components/app/app-shell.tsx` 现在会在账号态下自动刷新一次 `/user/account`，把最新 `profile`（含 `vipType`）回写到本地 store
- `src/lib/music/playability.ts` 现在做了更稳的兼容：
  - `userVipType > 0` 直接视为可播放 VIP 曲目
  - 已有账号态但 `vipType` 尚未同步回来时，不再过早误判成 `VIP Only`

用户已验证：

- 搜索页“大量歌曲被误标成 VIP”问题已修复
- 当前 VIP 账号下，搜索歌曲现在可以正常识别并播放

## 4. 当前已知问题 / 风险点

### 1. 用户名模式和账号模式仍然是两套 capability

当前已经拆成两类判断：

- 活跃会话：基于 `profile.userId` 即可成立
- 账号会话：必须有 `rawCookie` / `MUSIC_U`

这意味着后续开发时必须非常注意：

- 不能再把“是否已登录”简单等同于 cookie 是否存在
- `Library`、喜欢歌曲、公开歌单、私有收藏分区要分别判断
- `For You`、喜欢按钮、FM、收藏状态这类账号专属能力，用户名只读模式必须走清晰的降级路径

### 2. `liked songs` 的全量播放仍有批量拉取边界

当前实现已经比上一轮更接近旧版，但仍有边界要注意：

- 大歌单播放全部时，会基于 `trackIds` 分批调用歌曲详情接口
- 如果后端对某些 ID 返回缺失 / 限流 / 失败，最终构建的全量队列仍可能少歌
- 目前还没补“失败批次重试”与更强的错误提示

### 3. `Next Up` 和 Player 的行为已补齐不少，但视觉还没完全贴齐旧版

当前已完成的重点是行为层，但这些仍未完全对齐：

- 播放器按钮尺寸
- 图标状态
- 列表视觉密度
- 第三行文案节奏
- 队列页视觉细节
- 更多旧版 hover / active 反馈

### 3.5 当前 VIP 主链路已修复，但仍要注意“取流成功”与“UI 判定”是两套链路

当前已修：

- 客户端取流带本地账号 cookie
- 页面列表按当前账号态二次重算 `playable / reason`
- 账号态进入应用时自动补刷一次 `/user/account`

后续仍要注意：

- 只修页面上的 `VIP Only` 文案，不代表播放器一定拿到完整 URL
- 只修播放器取流，不代表列表页不会继续误标
- 这两条链路必须继续一起维护，不要只修其中一边

### 4. `Library` 还不是最终完整页

虽然当前已补：

- 收藏专辑
- 收藏艺人
- 最近播放
- 喜欢歌曲完整播放队列

但旧版还包括更多分区 / 能力：

- MV
- 云盘
- 更完整的 tab / 上下文菜单 / 播放模式细节

### 5. 来源语义仍未覆盖所有播放入口

当前已覆盖：

- 歌单
- 专辑
- 艺人
- 日推
- liked songs
- 私人 FM
- 搜索

但仍未完全覆盖：

- Explore 的不同分类上下文
- 排行榜 / 首页部分入口的更细粒度来源
- 更细的滚动位置或筛选态恢复

### 6. 构建阻塞仍存在

当前本地校验结果：

- `./node_modules/.bin/tsc --noEmit`：通过
- `bunx vitest run test/lib/player-track.test.ts test/stores/player-store.test.ts`：通过
- 更早阶段 `bunx vitest run`：通过
- `bun run build`：**未通过**

当前 `build` 阻塞不是最近几轮核心业务代码引入的，而是仓库既有问题：

- `src/routes/demo.i18n.tsx` 依赖的 `../logo.svg` 缺失
- 当前环境下 `wrangler` 试图写 `/root/.config/.wrangler/logs/*`，会遇到权限问题

因此：

- 不要把 `bun run build` 失败直接误判成最近几轮首页 / 播放器 / 搜索来源改动导致

## 5. 推荐下一步顺序

建议下一位开发者严格按下面顺序继续：

### Step 1：优先继续收 `Navbar / Player / 首页` 的视觉 parity

优先补：

- `RoutePlaceholder` 系页面骨架目前仍明显不像原版，优先继续拆掉“统一占位页”痕迹
- Navbar 尺寸、字重、hover 细节
- Player 按钮尺寸、图标状态、队列页样式
- 首页卡片间距、封面比例、标题字重
- `For You` 两张卡的视觉微调
- 播放器第三行来源文案节奏与按钮反馈

### Step 2：继续扩播放器来源语义

优先考虑：

- Explore 的分类来源
- 首页不同区块来源
- 排行榜 / 推荐歌单来源
- 更细粒度的搜索恢复（如滚动位置，如果后续确有必要）

### Step 3：继续扩 `Library`

优先考虑：

- 更完整的旧版信息结构
- tab / 分区切换
- 云盘 / MV 等剩余分区
- 更多旧版播放模式动作

### Step 4：最后再处理构建问题

在业务主线继续推进之前，不建议优先把大量时间花在：

- demo 页面资源缺失
- 当前环境里的 wrangler 日志权限

除非下一位开发者明确要以“恢复 `bun run build`”为目标单独收口。

## 6. 当前不要做的事情

- 不要把主线切回 `better-auth`
- 不要优先扩展 demo 页面或 `about` 页面
- 不要优先做非原版新增体验
- 不要在 parity 还没收口前大量引入默认 `shadcn/ui` 外观
- 不要把“代码写完”重新当成“已验证完成”
