## 4. 当前已知问题 / 风险点

### 1. 本轮大部分新增行为还缺人工回归

目前代码已经补到：

- 登录分流
- 用户名只读模式
- Library 扩展分区
- liked songs 全量播放队列
- `playNextQueue` 插队队列

但这些主要还是代码完成态，**尚未在真实浏览器里完成完整手测**。

### 2. 用户名模式和账号模式现在是两套 capability

当前已经拆成两类判断：

- 活跃会话：基于 `profile.userId` 即可成立
- 账号会话：必须有 `rawCookie` / `MUSIC_U`

这意味着后续开发时必须非常注意：

- 不能再把“是否已登录”简单等同于 cookie 是否存在
- `Library`、喜欢歌曲、公开歌单、私有收藏分区要分别判断

相关经验已写入：

- `docs/lessons-learned.md`

### 3. `liked songs` 的全量播放是分批详情拉取

当前实现已经比上一轮更接近旧版，但仍有边界要注意：

- 大歌单播放全部时，会基于 `trackIds` 分批调用歌曲详情接口
- 如果后端对某些 ID 返回缺失 / 限流 / 失败，最终构建的全量队列仍可能少歌
- 目前还没补“失败批次重试”与更强的错误提示

### 4. `Next Up` 三段结构已补，但视觉还没完全贴齐旧版

当前已完成行为层：

- 当前播放
- 插队播放
- 后续主队列

但仍未完全对齐的点包括：

- 列表视觉密度
- 图标细节
- 文案节奏
- 更多旧版上下文动作

### 5. `Library` 还不是最终完整页

虽然本轮已补：

- 收藏专辑
- 收藏艺人
- 最近播放

但旧版还包括更多分区 / 能力：

- MV
- 云盘
- 更完整的 tab / 上下文菜单 / 播放模式细节

### 6. 构建阻塞仍存在

当前本地校验结果：

- `bunx vitest run`：通过
- `./node_modules/.bin/tsc --noEmit`：通过
- `bun run build`：**未通过**

当前 `build` 阻塞不是本轮核心业务代码引入的，而是仓库既有问题：

- `src/routes/demo.i18n.tsx` 依赖的 `../logo.svg` 缺失
- 当前环境下 `wrangler` 试图写 `/root/.config/.wrangler/logs/*`，会遇到权限问题

因此：

- 不要把 `bun run build` 失败直接误判成这轮登录 / Library / 队列改动导致

## 5. 推荐下一步顺序

建议下一位开发者严格按下面顺序继续：

### Step 1：先做人工回归

优先手测：

1. `/login`
2. `/login/account`
3. `/login/username`
4. `/library`
5. `/library/liked-songs`
6. `/next`
7. 首页顶栏与播放器常驻壳体

重点确认：

- 登录分流是否稳定
- 用户名模式是否能稳定进入音乐库
- Library 新分区是否按账号态 / 非账号态正确展示
- “播放全部”后 `/next` 是否拿到完整队列
- “下一首”是否正确进入插队队列

### Step 2：根据手测结果修 bug

本轮最可能出现的回归点：

- 嵌套路由显示问题
- 用户名模式 capability 判断错误
- 喜欢歌曲大歌单补详情时的异常分支
- 插队队列与主队列切换边界

### Step 3：继续做 Navbar / Player / 首页细节 parity

优先补：

- Navbar 尺寸、字重、hover 细节
- Player 按钮尺寸、图标状态、队列页样式
- 首页卡片间距、封面比例、标题字重
- `For You` 接回真实 `DailyTracksCard` / `FMCard` 业务能力

### Step 4：继续扩 `Library`

优先考虑：

- 更完整的旧版信息结构
- tab / 分区切换
- 云盘 / MV 等剩余分区
- 更多旧版播放模式动作

### Step 5：最后再处理构建问题

在业务主线继续推进之前，不建议优先把大量时间花在：

- demo 页面资源缺失
- 当前环境里的 wrangler 日志权限

除非下一位开发者明确要以“恢复 `bun run build`”为目标单独收口。
