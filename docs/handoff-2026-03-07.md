## 4. 当前已知问题 / 风险点

### 1. 还没完全恢复旧版登录分流

当前 `/login` 已可用，但还没有进一步拆到旧版的：

- `/login/account`
- `/login/username`

也就是说：

- 功能链路已打通
- 但和旧版登录入口结构还不完全一致

### 2. `liked songs` 的“播放全部”仍然不是全量大歌单队列

当前实现：

- `/library/liked-songs` 支持分页显示
- 页面会按当前分页或已加载数据补拉歌曲详情
- 但“播放全部”现在主要基于**当前已加载曲目集**，并不是对超大喜欢歌单做一次完整全量队列构建

如果后续要更接近旧版，需要：

- 基于 `trackIds` 分批补齐完整歌曲详情
- 再构建完整播放队列

### 3. `Next Up` 还没有旧版 `playNextList` 插队队列

当前播放器只有一个主队列，没有旧版那种独立的“插队播放列表”。
因此：

- `/next` 现在是“当前主队列视图”
- 不是旧版完整的“当前播放 + 插队播放 + 下一首”三段结构

### 4. `Library` 仍是最小可用版

当前只恢复了最核心部分：

- 用户信息
- 喜欢的音乐
- 用户歌单列表

还没恢复旧版这些分区：

- 专辑
- 艺人
- MV
- 云盘
- 播放历史

### 5. 生成产物 / 全量检查问题仍存在

仓库仍存在已知阻塞：

- 缺少部分生成产物
- 全量 TypeScript 检查暂时不能作为完成标准

典型包括：

- `routeTree.gen.ts` 自动生成依赖
- Paraglide 运行时代码

## 5. 推荐下一步顺序

建议严格按下面顺序继续：

### Step 1：补旧版登录分流页

优先做：

- `/login/account`
- `/login/username`
- `/login` 作为旧版分流入口页恢复

原因：

- 当前虽然“能登录”，但还不够接近旧版路径
- 这是最明显的一块 parity 缺口

### Step 2：继续补 `Library` 结构

优先补：

- 收藏专辑
- 收藏艺人
- 播放历史

原因：

- 当前用户已经能登录
- `Library` 是下一块最自然能继续扩展的区域

### Step 3：继续完善 `liked songs`

优先补：

- 完整“播放全部”全量队列
- 更接近旧版歌单详情页样式
- 搜索 / 过滤（如果要贴近旧版 liked songs 行为）

### Step 4：再回到播放器增强

优先补：

- 喜欢歌曲按钮
- 列表来源跳转
- `playNextList` 插队队列

### Step 5：最后再考虑 MV

当前用户已明确说：**先不做 MV**。
因此 MV 相关工作现在不要抢主线。

## 6. 当前不要做的事情

- 不要把主线切回 `better-auth`
- 不要优先扩展 demo 页面或 about 页面
- 不要优先做非原版新增体验
- 不要在 parity 还没收口前大量引入默认 `shadcn/ui` 外观
- 不要把“能跑”重新当成“已完成”
